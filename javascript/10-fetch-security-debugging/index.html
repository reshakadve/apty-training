Fetch API and Async/Await

Fetch API

The Fetch API is a modern interface for making HTTP requests. It replaces the older XMLHttpRequest and uses Promises for easier syntax.

Basic Fetch Example

fetch('https://jsonplaceholder.typicode.com/posts/1')
.then(response => response.json())
.then(data => console.log(data))
.catch(error => console.error('Error:', error));

POST Request Example

fetch('https://jsonplaceholder.typicode.com/posts', {
method: 'POST',
headers: { 'Content-Type': 'application/json' },
body: JSON.stringify({ title: 'Hello World', body: 'Post body', userId: 1 })
})
.then(response => response.json())
.then(data => console.log(data))
.catch(error => console.error('Error:', error));

Fetching with Headers

fetch('https://api.example.com/data', {
method: 'GET',
headers: {
'Authorization': 'Bearer YOUR_TOKEN',
'Accept': 'application/json'
}
})
.then(response => response.json())
.then(data => console.log(data))
.catch(error => console.error('Error:', error));
Async/Await

Async/Await provides a cleaner, more synchronous way to handle asynchronous operations using Promises.

Basic Async/Await Example

async function fetchData() {
try {
const response = await fetch('https://jsonplaceholder.typicode.com/posts/1');
const data = await response.json();
console.log(data);
} catch (error) {
console.error('Error:', error);
}
}
fetchData();

POST Request with Async/Await

async function createPost() {
try {
const response = await fetch('https://jsonplaceholder.typicode.com/posts', {
method: 'POST',
headers: { 'Content-Type': 'application/json' },
body: JSON.stringify({ title: 'Async/Await Example', body: 'This is async', userId: 1 })
});
const data = await response.json();
console.log('Post created:', data);
} catch (error) {
console.error('Error creating post:', error);
}
}
createPost();
Combining Fetch API with Async/Await

Async/Await makes handling multiple fetch requests cleaner and easier to follow.

Fetching Multiple Resources

async function fetchPostsAndComments() {
try {
const postsResponse = await fetch('https://jsonplaceholder.typicode.com/posts');
const posts = await postsResponse.json();
const commentsResponse = await fetch('https://jsonplaceholder.typicode.com/comments');
const comments = await commentsResponse.json();
console.log('Posts:', posts);
console.log('Comments:', comments);
} catch (error) {
console.error('Error fetching data:', error);
}
}
fetchPostsAndComments();
Error Handling in Fetch with Async/Await

Use try...catch for robust error handling in async/await functions.

async function fetchWithErrorHandling() {
try {
const response = await fetch('https://jsonplaceholder.typicode.com/invalid-endpoint');
if (!response.ok) {
throw new Error(`HTTP Error: ${response.status}`);
}
const data = await response.json();
console.log(data);
} catch (error) {
console.error('Fetch error:', error.message);
}
}
fetchWithErrorHandling();

Security

Understanding and Preventing XSS (Cross-Site Scripting)

What is XSS?

Cross-Site Scripting (XSS) is a vulnerability that allows attackers to inject malicious scripts into webpages that are viewed by other users. The injected scripts run in the context of the user's browser, which can lead to data theft (like stealing cookies or session tokens), performing malicious actions (like submitting forms without the user's knowledge), or delivering malware to users.

Types of XSS
  • Reflected XSS: The malicious script is reflected off a web server and executed immediately after being injected by the attacker. This usually happens via URLs.
  • Stored XSS: The malicious script is stored on the server (in a database, for example) and is executed every time the data is retrieved and rendered by the browser.
  • DOM-based XSS: This occurs when JavaScript in the browser dynamically modifies the DOM (Document Object Model) based on untrusted input, allowing the attackerâ€™s script to run.
How to Prevent XSS
  • Escape User Input: Always escape special characters in user input before displaying them on a webpage to prevent them from being interpreted as code.
  • Use Safe APIs: Avoid using eval(), setTimeout() or setInterval() with string inputs. These can execute arbitrary code, creating a security risk.
  • Sanitize Input: Use libraries such as DOMPurify to sanitize user input before it is inserted into the DOM.
  • Set Content Security Policy (CSP): Implement a CSP header to restrict which sources can provide scripts to the page, reducing the risk of malicious content being executed from unauthorized sources.
  • Use HttpOnly and Secure Cookies: Ensure session cookies are flagged as HttpOnly and Secure to prevent them from being accessed via JavaScript, thus protecting them from being stolen via XSS.

Content Security Policy (CSP)

What is CSP?

Content Security Policy (CSP) is a security feature that allows you to control the sources from which a webpage can load resources, including scripts, styles, images, and more. It helps to prevent attacks like XSS by restricting the domains from which content can be loaded.

Basic CSP Example 

        Content-Security-Policy: default-src 'self'; script-src 'self' https://apis.example.com;

This CSP policy does the following:

  • default-src 'self': Allows resources (like images, stylesheets, etc.) to be loaded only from the same domain as the page.
  • script-src 'self' https://apis.example.com: Allows scripts to be loaded from the same domain and apis.example.com. No scripts can be loaded from other sources.
Key Directives in CSP
  • default-src: Specifies the default policy for loading content (used for most types of content).
  • script-src: Restricts where JavaScript can be loaded from.
  • style-src: Restricts where stylesheets can be loaded from.
  • img-src: Restricts where images can be loaded from.
  • connect-src: Restricts where XMLHttpRequest (AJAX requests), WebSockets, and EventSource connections can be made to.
  • frame-src: Specifies from where iframes can be embedded.
  • object-src: Specifies allowed sources for Flash, Java, and other plugins.
  • report-uri: Defines where violation reports should be sent if the policy is violated.
Benefits of CSP
  • Mitigates XSS: By restricting the sources of JavaScript and other resources, CSP can help prevent the execution of malicious scripts.
  • Limits Resource Injection: Even if an attacker tries to inject scripts, CSP can block them from being executed if they are loaded from untrusted sources.
  • Granular Control: You can specify exactly which sources are trusted, providing fine-grained control over what can be loaded and executed on your pages.
How to Implement CSP
  • Add the CSP header in your server configuration or in your HTML <meta> tag.
  • Server-side (e.g., Apache): 
    Header set Content-Security-Policy "default-src 'self'; script-src 'self';"
  • Using the meta tag in HTML (not recommended for complex policies): 
    <meta http-equiv="Content-Security-Policy" content="default-src 'self';">

Preventing CSRF (Cross-Site Request Forgery)

What is CSRF?

Cross-Site Request Forgery (CSRF) is an attack where a malicious website tricks a user into performing an action (such as changing a password or making a purchase) on another website where the user is authenticated. The attacker exploits the user's authentication (usually by sending a request with the user's credentials or cookies).

How to Prevent CSRF
  • Anti-CSRF Tokens: The most common way to prevent CSRF is by using anti-CSRF tokens. These tokens are unique for each request and are embedded in each form or AJAX request. The server then checks the token to ensure that the request is legitimate.
  • SameSite Cookies: The SameSite cookie attribute helps prevent CSRF by restricting when cookies are sent with cross-site requests. If you set SameSite=Lax or SameSite=Strict, the browser will only send cookies when the request originates from the same site.
  • Custom Headers: When making AJAX requests, use custom headers that are not included automatically by browsers with cross-origin requests. The server can then check for the presence of these headers to validate the request.
  • Use Secure HTTP Methods: CSRF attacks typically exploit the use of unsafe HTTP methods such as GET for state-changing actions (like submitting a form). Always use secure methods like POST, PUT, DELETE for actions that modify data or change the state on the server.
  • Same-Origin Policy: The Same-Origin Policy (SOP) ensures that a webpage can only make requests to the same domain that served the page. This prevents attackers from sending unauthorized requests from a different domain using the user's cookies.

JavaScript Debugging Techniques and Tools

Debugging JavaScript directly in the browser is an essential skill for developers. Modern browsers come equipped with powerful Developer Tools (DevTools) that enable you to troubleshoot and debug JavaScript code. Below, we'll walk you through various debugging techniques and tools available in browser environments, particularly focusing on the Google Chrome Developer Tools.

1. Console Logging with console.log(), console.error(), and console.warn()

Purpose: The most basic and common debugging technique. By logging variables, errors, and warnings to the browser's console, you can easily inspect your code at runtime.

How to Use:

  • console.log(): Prints general information.
  • console.error(): Logs errors with a red-colored output.
  • console.warn(): Logs warnings with a yellow-colored output.

Example:


let number = 10;
console.log("Current number:", number); // Logs to console
console.error("An error occurred"); // Logs as error
console.warn("This is a warning!"); // Logs as warning
2. Setting Breakpoints in the Source Code

Purpose: A breakpoint allows you to pause code execution at a specific line, so you can inspect the state of the application (variables, call stack, etc.).

How to Use:

  • Open DevTools (Ctrl + Shift + I or F12).
  • Go to the Sources tab.
  • Find the JavaScript file you want to debug.
  • Click on the line number where you want to set the breakpoint.

Example: After setting a breakpoint in your code, the execution will pause at that line, allowing you to inspect the value of variables and step through the code.

3. Using the debugger Statement

Purpose: The debugger statement pauses code execution at the point where it's invoked. It works just like a breakpoint in DevTools.

How to Use:

  • Insert the debugger statement in your code where you want to pause execution.
  • When the code hits the debugger statement, it will pause in DevTools.

Example:


function testFunction() {
  let a = 5;
  debugger; // Execution will pause here
  let b = 10;
  console.log(a + b);
}
testFunction();
4. Stepping Through Code

Purpose: Step through code to see exactly how it executes. This helps you understand the flow of control and inspect variable values at each step.

How to Use:

  • Once a breakpoint is hit or the debugger statement is encountered, you can use DevTools controls to step through the code.
  • Step over (F10): Executes the current line and moves to the next line.
  • Step into (F11): Enters any function calls that are on the current line.
  • Step out (Shift + F11): Skips out of the current function and moves to the calling code.
5. Inspecting Variables and Call Stack

Purpose: Inspect the state of your application while debugging. You can examine local variables, global variables, and the call stack (how functions are called).

How to Use:

  • In DevTools, once the code execution pauses (via a breakpoint or debugger), you can view variables in the Scope section.
  • The Call Stack panel shows the sequence of function calls that led to the current line of code.

Example: The Scope section shows local variables, and the Call Stack shows the functions in the current execution context.

6. Console and Watch Expressions

Purpose: Evaluate expressions and inspect variables in real-time while debugging.

How to Use:

  • In DevTools, go to the Console tab and interact with the application by typing JavaScript expressions directly in the console.
  • Watch Expressions allow you to add specific expressions you want to monitor throughout debugging.

Example: In the Watch panel, you can add an expression like myArray.length to see its value during debugging.

7. Network Tab to Monitor Requests

Purpose: The Network tab allows you to inspect network requests made by your JavaScript code (such as API calls).

How to Use:

  • Open DevTools and go to the Network tab.
  • Reload the page or trigger the network request.
  • Inspect details of HTTP requests such as URL, method, status code, response body, and headers.

Example: When making an API call with fetch, you can monitor the request and response in the Network tab.

8. Monitoring Events (Event Listeners)

Purpose: Monitor specific events, such as clicks, key presses, or window resizing.

How to Use:

  • In the Elements tab of DevTools, right-click on an element and select Inspect.
  • Go to the Event Listeners tab, and you can see all events attached to the element (like click, resize, keypress).

Example: You can break on a specific event listener, such as a click event, to see how the event is handled.

Browser Developer Tools: Debugging Tabs
1. Elements Tab

What it is: This tab lets you inspect and manipulate the DOM and CSS of a webpage.

How to Use: You can click on an element in the page and view its HTML structure. You can also edit HTML and CSS live to test changes.

2. Console Tab

What it is: The Console tab is where you can log output, run JavaScript commands, and view errors, warnings, and logs from your code.

How to Use: You can interact with your page using JavaScript directly in the console. It also displays console.log() messages, errors, and warnings.

3. Sources Tab

What it is: The Sources tab is where you can view, edit, and set breakpoints in the JavaScript files loaded by the browser.

How to Use: You can set breakpoints, step through your code, and inspect variables during execution.

4. Network Tab

What it is: The Network tab lets you monitor all the network requests made by your page, including API calls, image requests, and scripts.

How to Use: You can view request details, such as HTTP status codes, request headers, and response data.

5. Performance Tab

What it is: The Performance tab helps you analyze the runtime performance of your page.

How to Use: You can record and view a timeline of page rendering and script execution.
